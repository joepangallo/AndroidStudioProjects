The one idea

A screen is a key. The back stack is a list of keys.

Forget Activities and Intents. In Nav3, everything you see is driven by a single observable list. Whatever key sits on top of that list is the screen on screen right now.

Navigating forward = add a key to the list.

Going back = remove the top key.

A key doesn't just name a screen — it carries that screen's arguments. DetailKey(message) replaces what an Intent extra used to do.

The four moving parts

What actually makes it work

The entire navigation system in this app is four cooperating pieces. Click any card to expand the real code from MainActivity.kt.

1

NavKey — the identity of a screen

A marker interface. Each key type is one screen and holds that screen's arguments.

▸

HomeKey takes no arguments, so it's a single shared data object. DetailKey carries a message, so it's a data class. @Serializable lets Nav3 save & restore keys across rotation and process death.

@Serializable
data object HomeKey : NavKey

@Serializable
data class DetailKey(val message: String) : NavKey
2

rememberNavBackStack — the list itself

Creates the observable back stack, seeded with the start screen, surviving recomposition & rotation.

▸

It returns a mutable, observable list of NavKeys. You mutate it with plain list operations — .add(...) and .removeLastOrNull() — and the UI follows.

val backStack = rememberNavBackStack(HomeKey)
// starts as: [HomeKey]  → Home is shown
3

NavDisplay — the renderer

Observes the back stack, draws the screen for the top key, and handles system / predictive back.

▸

Give it the backStack, an onBack that pops, and an entryProvider that maps keys to UI. It animates transitions automatically when the list changes.

NavDisplay(
    backStack = backStack,
    onBack = { backStack.removeLastOrNull() },
    entryProvider = entryProvider { /* … */ }
)
4

entryProvider — key → screen map

A DSL that says "for this key type, draw this composable." Arguments are read straight off the key.

▸

No bundles, no getStringExtra — the message is just a typed property on the key. The compiler guarantees it's there.

entryProvider {
    entry<HomeKey> {
        HomeScreen(onOpenDetail = {
            backStack.add(DetailKey("Hello from Home!"))
        })
    }
    entry<DetailKey> { key ->
        DetailScreen(message = key.message, onBack = {
            backStack.removeLastOrNull()
        })
    }
}

The real source

Annotated MainActivity.kt

The complete navigation code, exactly as it lives in the project. Click any highlighted line to learn what it does.

① Keys ② AppNavigation ③ Screens 
// A key identifies a screen AND carries its arguments.
@Serializable                       // generates a serializer for this key
data object HomeKey : NavKey      // no args → one shared object

@Serializable
data class DetailKey(val message: String) : NavKey  // message = its argument
@Composable
fun AppNavigation(modifier: Modifier = Modifier) {
    val backStack = rememberNavBackStack(HomeKey)   // the back stack, starts at Home

    NavDisplay(
        backStack = backStack,                  // the list it observes
        modifier = modifier,
        onBack = { backStack.removeLastOrNull() }, // predictive/system back → pop
        entryProvider = entryProvider {           // map key type → screen UI
            entry<HomeKey> {
                HomeScreen(onOpenDetail = {
                    backStack.add(DetailKey("Hello from Home!")) // push = go forward
                })
            }
            entry<DetailKey> { key ->            // receive the top key as `key`
                DetailScreen(message = key.message, // read arg straight off the key
                    onBack = { backStack.removeLastOrNull() })
            }
        }
    )
}
// Screens are ordinary composables — they don't know about navigation.
// They only receive callbacks (onOpenDetail / onBack) and data (message).
@Composable
fun HomeScreen(onOpenDetail: () -> Unit, modifier: Modifier = Modifier) {
    Column(modifier = modifier) {
        Text(text = "Hello Android!")
        Button(onClick = onOpenDetail) { Text("Open Detail") } // fires the push
    }
}

fun DetailScreen(message: String, onBack: () -> Unit, …) {
    Column {
        Text(text = message)                 // the argument carried by the key
        Button(onClick = onBack) { Text("Back") } // fires the pop
    }
}

Step by step

What happens when you tap "Open Detail"

One user tap sets off a precise chain of events. Step through it.

1

User taps the button

The Button's onClick runs onOpenDetail, the lambda Home was handed.

2

A new key is pushed

backStack.add(DetailKey("Hello from Home!")) — the list goes from [HomeKey] to [HomeKey, DetailKey].

3

NavDisplay notices the change

The back stack is observable. Its top key is now a DetailKey, so NavDisplay recomposes.

4

entryProvider resolves the key

entry<DetailKey> matches. The key is handed in as key, and key.message is read.

5

DetailScreen animates in

DetailScreen(message = "Hello from Home!") renders and slides in. Back now pops to return Home.

Why this, not Intents

Nav3 vs. the old way

The project is named Intent for a reason — it demonstrates replacing the classic Activity-and-Intent model with a single-Activity, Compose-first back stack.

ConcernOld: Activities + IntentsNav3 (this app)
A screen is… a separate Activity launched by the OS a composable mapped to a NavKey
Going forward startActivity(Intent(...)) backStack.add(DetailKey(...))
Passing arguments putExtra("msg", …) → getStringExtra("msg") (untyped, can be null) DetailKey(message) — a typed, compiler-checked property
Going back OS pops the Activity task stack backStack.removeLastOrNull() — you own the list
State across rotation Activity recreated; save/restore bundles by hand @Serializable keys are saved & restored automatically
The back stack is… opaque, managed by the framework a plain observable list you can read, log, and mutate

Wiring it up

The dependencies that enable it

From app/build.gradle.kts and the version catalog — three additions turn a plain Compose app into a Nav3 app.

// app/build.gradle.kts — dependencies
implementation(libs.androidx.navigation3.runtime)   // NavKey, rememberNavBackStack, entryProvider
implementation(libs.androidx.navigation3.ui)        // NavDisplay
implementation(libs.kotlinx.serialization.json)     // saves @Serializable keys

// app/build.gradle.kts — plugins
alias(libs.plugins.kotlin.serialization)            // generates the key serializers

// gradle/libs.versions.toml
nav3 = "1.0.1"      composeBom = "2026.02.01"      kotlin = "2.2.10"
